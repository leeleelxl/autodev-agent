# Baseline 对比实验设计

## 目标

通过与现有系统对比，证明 AutoDev 的创新价值和性能优势。

## 对比对象

### 1. GPT Engineer（主要 Baseline）
- **类型**: 单 Agent 架构
- **GitHub**: https://github.com/AntonOsika/gpt-engineer
- **特点**: 简单的迭代生成，无多 Agent 协作
- **对比维度**: 架构复杂度、代码质量、成功率

### 2. 单一 LLM（基础 Baseline）
- **类型**: 直接使用 Claude/GPT-4
- **方式**: 一次性生成代码
- **对比维度**: 证明多 Agent 协作的价值

### 3. MetaGPT（可选）
- **类型**: 多 Agent 架构
- **对比维度**: 协作机制的差异

## 测试数据集

### 方案 1: 自定义测试集（推荐）

创建 10-20 个不同难度的编程任务：

**简单任务（5个）**
1. 计算器应用
2. TODO 列表
3. 文件读写工具
4. 简单 HTTP 服务器
5. 数据排序算法

**中等任务（5个）**
1. REST API 服务
2. 数据库 CRUD 应用
3. 命令行工具（带参数解析）
4. 简单爬虫
5. 数据可视化工具

**复杂任务（5个）**
1. 用户认证系统
2. 实时聊天应用
3. 任务调度系统
4. 数据分析管道
5. 微服务架构示例

### 方案 2: SWE-bench 子集

从 SWE-bench 选取 20-50 个中等难度任务。

**优点**: 业界标准，可信度高
**缺点**: 任务复杂，需要更多时间

## 评估指标

### 1. 任务完成率
- **Pass@1**: 首次尝试成功率
- **Pass@3**: 3 次尝试内成功率
- **定义**: 代码能运行且通过基本测试

### 2. 代码质量
- **Pylint 分数**: 0-10 分
- **复杂度**: 圈复杂度
- **可读性**: 代码行数、注释率

### 3. 测试覆盖率
- **单元测试覆盖**: 百分比
- **测试通过率**: 生成的测试是否通过

### 4. 效率指标
- **Token 使用量**: 总 token 数
- **执行时间**: 完成任务的时间
- **迭代次数**: 需要几次改进

### 5. 错误分析
- **语法错误**: 代码是否能解析
- **运行时错误**: 执行时的错误
- **逻辑错误**: 功能不符合预期

## 实验流程

### 阶段 1: 环境准备
1. 安装 GPT Engineer
2. 准备测试任务列表
3. 配置评估脚本

### 阶段 2: 数据收集
对每个任务：
1. AutoDev 执行 3 次
2. GPT Engineer 执行 3 次
3. 单一 LLM 执行 3 次
4. 记录所有指标

### 阶段 3: 数据分析
1. 计算平均值和标准差
2. 生成对比图表
3. 统计显著性检验

### 阶段 4: 报告撰写
1. 实验设置说明
2. 结果展示（表格+图表）
3. 分析和讨论
4. 结论

## 预期结果

### AutoDev 优势
- **代码质量**: +15-20% vs GPT Engineer
- **测试覆盖**: +25-30% vs 单一 LLM
- **首次成功率**: +10-15% vs GPT Engineer
- **Token 效率**: 相当或略高（因为多次调用）

### 创新点展示
1. **多 Agent 协作**: 分工明确，质量更高
2. **自动测试**: 生成并执行测试
3. **代码审查**: 自动发现问题并改进
4. **记忆学习**: 从历史经验中学习

## 时间估算

- **环境准备**: 1-2 天
- **数据收集**: 3-5 天（20 任务 × 3 系统 × 3 次）
- **数据分析**: 1-2 天
- **报告撰写**: 1-2 天

**总计**: 1-2 周
