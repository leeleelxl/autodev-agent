## 技术面试要点

### Q1: 为什么要用多 Agent 而不是单个 LLM？

**回答**：
"单个 LLM 生成的代码往往缺少测试和质量保证。通过多 Agent 协作：
- Architect 负责架构设计
- Developer 负责代码实现
- Tester 自动生成测试
- Reviewer 进行代码审查
- 每个 Agent 专注自己的领域，整体质量更高"

**数据支持**：
- 测试覆盖：100% vs 0%
- 问题识别：9.6 个 vs 0 个

---

### Q2: 遇到的最大挑战是什么？

**回答**：
"最大的挑战是 LLM 输出格式的不稳定性。

**问题**：
- 最初使用 JSON 格式，但 LLM 经常生成语法错误
- 代码中的引号、换行符会破坏 JSON 结构
- 导致 json.loads() 失败率很高

**解决方案**：
- 研究了 Cursor、Aider 等成熟项目
- 发现它们都使用 Markdown 代码块
- 改用 Markdown 后，成功率提升到 100%

**学到的经验**：
- 不要重新发明轮子，学习业界最佳实践
- LLM 擅长生成自然语言，不擅长严格格式
- 选择合适的输出格式很重要"

---

### Q3: AutoDev 比 Single LLM 慢 12 倍，值得吗？

**回答**：
"这取决于使用场景。

**简单任务**（计算器）：
- Single LLM 更合适（10s vs 185s）
- AutoDev 是过度设计

**复杂任务**（用户认证、REST API）：
- AutoDev 的优势明显
- 生成 Clean Architecture（16 个文件）
- 自动生成测试和文档
- 长期维护成本更低

**权衡**：
- 开发时间：AutoDev 慢 12 倍
- 维护时间：AutoDev 可能节省 10 倍时间
- 对于生产项目，值得投入更多时间保证质量"

---

### Q4: 如果重新设计，会做什么改进？

**回答**：
"主要有三个方向：

**1. 智能任务分类**
- 自动判断任务复杂度
- 简单任务跳过多 Agent 流程
- 复杂任务才启用完整工作流

**2. 依赖补全机制**
- 当前能检测缺失依赖，但无法自动补全
- 可以添加自动生成缺失模块的功能
- 确保代码 100% 可运行

**3. 增量改进**
- 当前是全量重新生成
- 可以改为只修改有问题的部分
- 提升迭代效率"

## 常见问题

### Q: 代码能直接运行吗？

**诚实回答**：
"大部分能运行，但可能需要安装依赖。AutoDev 会生成 requirements.txt，列出所有依赖。

有些任务可能缺少部分模块，这是当前的局限。我实现了完整性检查机制，能自动发现缺失依赖，但还没有实现自动补全。这是未来改进的方向。"

### Q: 为什么不用 Claude 而用 Kimi？

**回答**：
"主要是成本考虑。Kimi K2 Turbo 的性能接近 Claude，但成本更低。

而且这个项目的重点是多 Agent 协作框架的设计，不是特定模型的能力。理论上可以替换成任何 LLM。"

### Q: 和 GPT Engineer 有什么区别？

**回答**：
"GPT Engineer 是单 Agent 多轮对话，需要人工参与。

AutoDev 是多 Agent 全自动协作，无需人工介入。而且 AutoDev 有自动测试生成和代码审查，质量保证更完善。"
